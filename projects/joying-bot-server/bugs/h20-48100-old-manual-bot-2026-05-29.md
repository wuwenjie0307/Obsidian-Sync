# h20 48100 still pointed to old manual Bot

## 问题描述

`test` 分支和 Jenkins 部署目录 `/data/project/test_ai_botserver` 已经包含新调度链路代码，但外部公网入口 `223.112.222.90:48100` 实际映射到 h20 内部 `8100`。检查发现 `8100` 仍是旧手动目录 `/data/projects/joyingbot-new` 启动的进程，导致外部 CRM 如果使用 48100，会打到旧代码。

## 复现步骤

1. 在 h20 查看 `app_server_api.py` 进程工作目录。
2. 发现 `8100` 对应 `/data/projects/joyingbot-new`，而 Jenkins 新代码运行在 `8017` 的 `/data/project/test_ai_botserver.20260529193010`。
3. 调用 `POST http://127.0.0.1:8100/crm/submit_heygem_whisper_video_task`，返回旧接口的缺参 400，而不是新代码的 410。

## 期望行为

外部 `223.112.222.90:48100` 应进入当前测试服新代码，至少满足：

- `/status/check` 返回 ok。
- `/crm/submit_heygem_whisper_video_task` 返回 410。
- `/crm/generate_video_task` 使用新调度链路。
- `/crm/voice_clone_audition` 使用新的音色参数范围。

## 实际行为

48100 对应的 8100 原本仍在旧手动目录运行，代码缺少：

- `router/service/video_server2/voice_params.py`
- `VideoGenerateTask` 三个音色字段模型定义
- `/submit_heygem_whisper_video_task` 410 禁用逻辑
- 调度链路传递音色参数逻辑

## 环境信息

- 分支: `test`
- h20 Jenkins 部署目录: `/data/project/test_ai_botserver.20260529193010`
- 旧手动目录: `/data/projects/joyingbot-new`
- 公网入口: `223.112.222.90:48100 -> h20:8100`

## 修复方案

已停止 `/data/projects/joyingbot-new` 下监听 8100 的旧手动 Bot 进程，并从 `/data/project/test_ai_botserver` 当前 Jenkins 部署目录重新启动 8100：

```bash
cd /data/project/test_ai_botserver
TZ=Asia/Shanghai nohup /data/server/anaconda3/envs/botserver/bin/python app_server_api.py --env=dev --jobStatus=false --port=8100 > /tmp/bot_8100_test_ai_botserver.log 2>&1 &
```

修复后从跳板机访问 `223.112.222.90:48100` 验证：

- `/status/check` 返回 200。
- `/crm/submit_heygem_whisper_video_task` 返回 410。
- `/crm/generate_video_task` 空参数返回 400，提示 `job_id 不能为空`。
- `/crm/voice_clone_audition` 空参数返回 400，提示 `voice_file_url 不能为空`。

## 优化点

- 当前 8100 是手动 nohup 进程，不归 supervisor 管理。后续建议新增独立 supervisor program 或让网络映射直接指向 supervisor 的 8017，避免 Jenkins 部署后 48100 再次落到旧代码。
- `app_config` 启动日志会打印完整配置，包含敏感信息，应后续改为脱敏输出。
