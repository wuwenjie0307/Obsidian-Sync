# 正式服 6014 口型 Docker 被 8405 卡住导致 busy

## 问题描述

2026-06-04 正式服视频合成出现“任务提交失败 / 忙碌中 / 视频合成失败”。根据生产 `run.log` 排查手册，先从 `/data/project/prod_ai_autodone/logs/run.log` 确认业务进程状态，再定位到旧口型 Docker 端口 `6014`，对应容器 `duix-avatar-gen-video14`。

## 复现步骤

1. 正式服有视频任务提交到 `6014` 的 `/easy/submit`。
2. `6014` 正在处理一个长时间未结束的任务。
3. 后续任务继续提交到同一端口，容器返回 busy，上层记录为“任务提交失败 / 忙碌中”。

## 期望行为

旧口型容器长时间处理任务时，上层应避开 busy 实例，换用空闲实例、退避重试，或明确把任务留在队列等待；不应让后续任务直接进入失败态。

## 实际行为

`duix-avatar-gen-video14` 在处理任务 `8405` 时卡住，期间多次新任务提交到 `6014` 被容器判定 busy。

关键时间线：

- `2026-06-04 10:48:27`：`6014` 收到 `/easy/submit`。
- `2026-06-04 10:48:37`：任务 `8405` 开始执行。
- `2026-06-04 13:01:42` 到 `13:12:30`：`8457 / 8461 / 8464 / 8466 / 8467 / 8469 / 8471 / 8472 / 8473 / 8474 / 8475 / 8479` 等提交被记录为 `busy`。
- `2026-06-04 14:24:53` 到 `14:47:04`：`8494 / 8497 / 8499 / 8502 / 8505 / 8508` 等继续被记录为 `busy`。
- `2026-06-04 14:48:42`：容器日志出现 `8405 -> task get result timeout, process will restart`。
- `2026-06-04 14:48:54`：`8405 -> 耗时:14426.513s`，约 4 小时。

## 环境信息

- 应用进程：`ai_autodone_py_prod`
- 应用日志：`/data/project/prod_ai_autodone/logs/run.log`

- 宿主机端口：`6014 -> 8383/tcp`

## 原因

直接原因是旧口型 Docker `6014` 是单任务实例，任务 `8405` 长时间未正常产出结果，容器内部一直处于 busy 状态。后续任务继续提交到同一实例时，容器按 busy 返回，上层 Bot 将其记录为“任务提交失败 / 忙碌中 / 视频合成失败”。

本次没有看到生产主进程崩溃，也没有看到当前队列持续堆积；问题集中在 `6014` 容器内部任务超时。

## 解决方案

本次只做了只读排查，没有重启生产服务，也没有修改数据库。

现场处理结果：

- 确认 `ai_autodone_py_prod` 为 `RUNNING`。
- 确认当前 `run.log` 定时任务正常扫描，`待生成任务数: 0`。
- 确认 `duix-avatar-gen-video14` 当前仍运行。
- 确认卡住的 `8405` 已在容器内部超时后释放。
- 确认后续 `8510` 已正常生成成功，当前没有持续 busy。

如果下次仍处于持续 busy：

1. 先确认是哪一个 `600X/601X` 端口返回 busy。
2. 用 `docker ps` 映射到具体 `duix-avatar-*` 容器。
3. 查容器日志里当前正在跑的任务 code。
4. 如果持续超过业务允许时间，先评估影响，再人工重启对应容器或把该实例从调度中摘除。
5. 对已经失败的业务任务，结合 DB 状态决定是否人工重跑。

## 优化点

- 上层调度识别 `code=10001` / `msg=忙碌中` 后，应换用其他空闲实例或退避重试，不要直接把任务判失败。
- 增加旧口型 Docker 的 busy 次数监控；同一容器连续 busy 超过阈值时告警。
- 增加 `task get result timeout` 关键字监控；出现后记录对应容器、任务 code 和耗时。
- 在业务日志中保存“业务 task_id / job_id / Docker code / Docker 端口”的映射，避免下次只能靠时间窗口反查。
- 对旧口型实例增加自动摘除或自恢复机制，避免单实例卡住拖累后续任务。
