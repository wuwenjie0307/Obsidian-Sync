# 音色克隆新增参数 CRM 测试方案

## 最新确认（2026-05-29）

结论：产品可以开始测试音色克隆新增参数，建议先测试听接口，不要先跑完整视频。

当前给 CRM/产品的联调入口：

```text
POST http://223.112.222.90:48100/crm/voice_clone_audition
```

本轮重点测试三个字段：

| 字段 | 建议测试值 | 备注 |
|---|---|---|
| `voice_emotion` | `1-8` | 先固定语速和音量，只切情绪 |
| `voice_speed` | `0.75`, `1.0`, `1.25`, `1.5`, `2.0`, `3.0` | `1.0` 为基准 |
| `voice_volume` | `25`, `50`, `75`, `100` | `50` 为基准，`100` 重点听是否爆音 |

执行建议：

1. CRM 先在测试环境把三个字段透传到 h20 Bot。
2. 产品用同一个参考音频 URL 和同一段文案做对比。
3. 每次只改一个字段，记录返回的 `data.voice_file_url`。
4. 先筛出音频效果较好的 2-3 组参数，再跑完整视频任务。

需要注意：

- `voice_file_url` 必须是 h20 能访问的音频 URL，最好是 CDN 地址。
- 产品或本地电脑直连 `223.112.222.90:48100` 可能超时，联调应以 CRM 测试服/后端机器访问为准。
- 如果 CRM 测试服访问 `48100` 超时，再找张晋良排查网络映射或访问策略。
- 不建议批量高并发测试；VoxCPM 当前按最多 2 个并发处理，LatentSync 是单并发。

- Bot 公网入口：`http://223.112.222.90:48100`

产品当前主要测试音色克隆新增三个字段的效果：

- `voice_emotion`
- `voice_speed`
- `voice_volume`

建议先测试听接口，不要一开始跑完整视频。试听接口可以快速隔离这三个参数对音频的影响。

## 测试接口

```text
POST http://223.112.222.90:48100/crm/voice_clone_audition
```

请求示例：

```bash
curl -s -X POST http://223.112.222.90:48100/crm/voice_clone_audition \
  -H "Content-Type: application/json" \
  -d '{
    "voice_file_url": "可被h20访问的参考音频URL",
    "text": "你好，这是本次音色克隆试听测试，请听一下语气、语速和音量是否自然。",
    "voice_emotion": 1,
    "voice_speed": 1.0,
    "voice_volume": 50
  }'
```

成功返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "voice_emotion": 1,
    "voice_speed": 1.0,
    "voice_volume": 50,
    "voice_file_url": "https://videos-test.joyingai.cn/xxx.wav"
  }
}
```

返回里的 `data.voice_file_url` 是产品要听的试听音频链接。

## 参数范围

| 字段 | 可测值 | 说明 |
|---|---:|---|
| `voice_emotion` | `1-8` | 情绪 |
| `voice_speed` | `0.75`, `1.0`, `1.25`, `1.5`, `2.0`, `3.0` | 语速，`1.0` 是默认 |
| `voice_volume` | `0-100` | 音量，`50` 是默认/原始音量 |

`voice_emotion` 口径：

| 值 | 含义 |
|---:|---|
| `1` | 正常 |
| `2` | 亲切 |
| `3` | 热情 |
| `4` | 激昂 |
| `5` | 严肃 |
| `6` | 喜悦 |
| `7` | 悲伤 |
| `8` | 愤怒 |

## 推荐测试方法

不要一上来排列组合全部测。建议分三轮，只改变一个变量。

### 第一轮：测情绪

固定：

```json
{
  "voice_speed": 1.0,
  "voice_volume": 50
}
```

依次测：

```text
voice_emotion = 1,2,3,4,5,6,7,8
```

目的：判断不同情绪有没有明显差异，哪些可用，哪些不自然。

### 第二轮：测语速

固定：

```json
{
  "voice_emotion": 1,
  "voice_volume": 50
}
```

依次测：

```text
voice_speed = 0.75, 1.0, 1.25, 1.5, 2.0, 3.0
```

目的：判断语速变化是否自然，尤其是 `2.0`、`3.0` 是否过快、发音是否糊。

### 第三轮：测音量

固定：

```json
{
  "voice_emotion": 1,
  "voice_speed": 1.0
}
```

建议先测：

```text
voice_volume = 25, 50, 75, 100
```

`50` 是基准。小于 50 是变小，大于 50 是变大。`100` 可能有爆音或失真风险，需要重点听。

## 测试记录表

建议产品按这个表记录：

| case | voice_file_url | text | emotion | speed | volume | 返回音频URL | 产品评分 | 问题备注 |
|---|---|---|---:|---:|---:|---|---:|---|
| baseline | 同一个音频 | 同一段文案 | 1 | 1.0 | 50 |  |  | 基准 |
| emotion_2 | 同一个音频 | 同一段文案 | 2 | 1.0 | 50 |  |  | 亲切 |
| emotion_3 | 同一个音频 | 同一段文案 | 3 | 1.0 | 50 |  |  | 热情 |
| speed_075 | 同一个音频 | 同一段文案 | 1 | 0.75 | 50 |  |  | 慢速 |
| speed_15 | 同一个音频 | 同一段文案 | 1 | 1.5 | 50 |  |  | 快速 |
| volume_75 | 同一个音频 | 同一段文案 | 1 | 1.0 | 75 |  |  | 偏大 |

产品主要听这些维度：

1. 音色像不像原音频。
2. 情绪变化是否明显。
3. 语速是否自然。
4. 音量是否过小、过大、爆音。
5. 是否有杂音、电音、断字、吞字。
6. 长文本是否稳定。

## 需要产品/CRM 提供什么

1. 参考音频 URL：`voice_file_url`
   - 必须是 h20 能访问的 URL。
   - 最好是 CDN URL。
   - 不要用本地文件路径。
   - 不要用浏览器 blob。
   - 不要用需要登录鉴权的内网页面地址。
   - 音频里最好是清晰人声，背景噪音少。

2. 固定测试文案：`text`
   - 第一轮建议用同一段短文案，方便对比参数差异。
   - 后面再测一段长文案，验证稳定性。

3. 参数组合表
   - 按 emotion、speed、volume 分轮测试。

4. 结果反馈
   - case 编号。
   - 返回音频 URL。
   - 可用/不可用。
   - 主观评分。
   - 问题描述。

## 联调分工

- CRM 后端/前端同事：负责把接口调通，或在页面上把三个字段传给 h20 Bot。
- 产品：负责听返回音频效果，给主观判断。
- 我们：负责确认参数是否到 VoxCPM、接口是否报错、模型链路是否正常。
- 张晋良：只在 CRM 测试服访问 `223.112.222.90:48100` 超时、不通时介入网络问题。

## 注意事项

1. 产品本机浏览器不一定能访问 `223.112.222.90:48100`。
   - 之前本地电脑访问超时，但跳板/生产侧访问通过。
   - 应让 CRM 测试服机器或 CRM 后端去调，不要以产品电脑直连为准。

2. 不要并发太多。
   - VoxCPM 当前最多并发 2 个。
   - 产品批量测参数时建议一个一个跑，或最多两个并发。

3. 先测试听，不要每个参数都跑完整视频。
   - 完整视频会进 LatentSync，较慢，且 LatentSync 是单并发。
   - 先把音频效果筛出来，再挑 2-3 组最满意的参数跑完整视频。

4. `voice_speed` 只能传允许值。
   - 支持：`0.75`, `1.0`, `1.25`, `1.5`, `2.0`, `3.0`
   - 不要传 `0.5`。

5. `voice_volume=50` 是基准。
   - 小于 50 变小。
   - 大于 50 变大。
   - `100` 可能有爆音风险。
