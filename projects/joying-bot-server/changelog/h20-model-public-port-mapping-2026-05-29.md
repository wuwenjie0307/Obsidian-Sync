## 最新确认

张晋良最新回复：

- `内部端口 8100、8101，对应的外部端口是 48100、48101`
- `在 h20 内部使用 8100、A800 那机器使用 48100 调用`

最终口径：

| 用途 | h20 内部地址 | 外部地址 |
|---|---|---|
| VoxCPM 音色克隆 API | `http://127.0.0.1:8100` | `http://223.112.222.90:48100` |
| LatentSync 唇形同步 API | `http://127.0.0.1:8101` | `http://223.112.222.90:48101` |
| Bot 主服务 `/crm/*` | `http://127.0.0.1:8018` | 暂未确认公网映射 |

## 影响

- h20 本机 Bot 调用模型服务时，继续使用 `127.0.0.1:8100/8101`。
- A800/生产服从外部调用 h20 模型服务时，使用 `223.112.222.90:48100/48101`。
- `48100/48101` 不是 Bot 主服务的公网入口，不能用于 CRM `/crm/*` 接口。
- 如果 CRM 测试服需要直接访问 h20 Bot `/crm/*`，仍需单独申请公网端口映射到 h20 内部 `172.16.220.119:8018`。

## 验证建议

在 A800/生产服上验证模型公网入口：

```bash
curl -s http://223.112.222.90:48100/health
curl -s http://223.112.222.90:48101/health
```

在 h20 本机验证内部模型入口：

```bash
curl -s http://127.0.0.1:8100/health
curl -s http://127.0.0.1:8101/health
```

在 h20 本机验证 Bot：

```bash
curl -s http://127.0.0.1:8018/status/check
```

## 2026-05-29 测试服目标澄清

伍文杰进一步澄清：当前目标不是生产服调用 h20 模型，也不需要生产服使用测试服模型；生产服有自己的生产模型方案。当前目标是 h20 测试服内完整跑通替换后的 Bot + VoxCPM + LatentSync 流程，并让测试服 Bot 的 `/crm/*` 服务能从外部访问。

在“不新增公网映射”的约束下，可复用现有公网 `48100` 映射，但需要调整 h20 内部端口占用：

- 让 Bot 对外占用已映射内部端口 `8100`，外部入口为 `223.112.222.90:48100`。
- 将 VoxCPM 从内部 `8100` 迁到一个新的 h20 内部端口，例如 `8110`，只供 Bot 本机调用。
- LatentSync 可继续使用内部 `8101`，Bot 本机调用 `127.0.0.1:8101`。
- h20 Bot 配置应改为：`server_port=8100`、`h20_api_base=http://127.0.0.1:8110`、`voxcpm_api_base=http://127.0.0.1:8110`、`latentsync_api_base=http://127.0.0.1:8101`。

风险：公网 `48100` 将不再是 VoxCPM API，而会变成 Bot `/crm/*` 入口；但当前目标明确为测试服 Bot 外部跑通，生产不依赖 h20 模型公网 API，因此该调整可接受。
