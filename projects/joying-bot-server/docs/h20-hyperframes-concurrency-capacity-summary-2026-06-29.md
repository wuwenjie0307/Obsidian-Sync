# H20 HyperFrames 7 并发资源压测总结

## 结论

H20 上 HyperFrames Docker runner 在单机、单 GPU0 可见条件下，使用约 72.58 秒视频任务做 7 并发压测，结果为 **7/7 成功**，无容器残留。

本次链路几乎不吃 GPU 计算，GPU0 平均/峰值利用率均为 `0.0%`。主要资源消耗在 CPU、Chromium、ffmpeg、磁盘 IO、容器调度等后处理环节。

从本次样本看，`HF_MAX_CONCURRENCY=7` 在 H20 测试服资源层面可行；正式环境建议灰度提升，并持续观察任务耗时、失败率、磁盘 IO、回调状态和容器残留。

## 资源换算

换算方式：

```text
100% CPU = 1 个 CPU 核
折合平均 CPU 核 = 平均 CPU% / 100
```

本次 7 并发：

```text
总平均 CPU 核 = 8.16 核
平均每路 CPU 核 = 8.16 / 7 = 1.17 核
H20 CPU 占比 = 8.16 / 224 = 3.642%
```

从 CPU 资源看，H20 余量很大。但不能简单按 `224 / 1.17` 线性推到上百并发，因为更高并发后可能先遇到：

- Chromium/浏览器实例调度开销
- ffmpeg 编码并发效率下降
- 磁盘 IO 和临时文件写入压力
- 容器启动和进程调度开销
- 文件句柄、共享内存、系统限制
- 回调和任务状态链路延迟

## 后续建议

1. H20 测试服 HyperFrames Docker runner 已验证 `7` 并发可成功跑完。
2. 本轮已控制为单 GPU0 可见，但实际 GPU 计算利用率为 `0%`，说明 HyperFrames 渲染瓶颈主要不是 H20 GPU 算力。
3. 7 并发平均只消耗约 `8.16` 个 CPU 核，占 H20 224 核约 `3.642%`，CPU 层面压力很小。
4. 可以在测试服将 `HF_MAX_CONCURRENCY` 提到 `7` 做真实任务观察。
5. 正式服建议灰度：先 `3` 或 `5`，观察稳定后再到 `7`。
6. 后续如果要继续摸上限，建议按 `10`、`14` 递增，并同时监控 load、磁盘 IO、容器失败率、任务耗时、回调状态和容器残留。

## 注意

本次压测是直接复制历史成功任务 manifest，启动 7 个临时 Docker 容器验证 runner 承载力；它不改数据库、不改真实任务、不重启 scheduler。

如果要验证线上完整链路，还需要通过 scheduler/CRM 真实提交 7 个任务，观察模型池、HyperFrames、上传、回调的端到端表现。
