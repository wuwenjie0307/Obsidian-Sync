# H20 混剪倍速字幕排查今日总结

## 今日工作总结

今天围绕产品反馈的“音色克隆倍速在混剪视频中导致字幕跟不上说话速度”进行了排查。

样例视频：

`https://videos-test.joyingai.cn/video/crm/20260608/user4_1780913803580_cd97a6a907897907.mp4`

完成事项：

1. 通过测试库 `zhugedata_test` 反查到了样例视频对应任务。
2. 确认任务为 `id=1350`、`job_id=1156`、`task_id=1138`，状态已成功，回调已完成。
3. 确认该任务使用了音色克隆高倍速：`voice_speed=3.0`、`voice_emotion=8`、`voice_volume=77`。
4. 确认该任务确实是混剪任务：本地素材表存在 1 条视频混剪素材，`is_mix_material=1`。
5. 确认混剪素材本身时长为 `13` 秒，但绑定文案约 `165` 字，属于短素材承载长文案的组合。
6. 追踪了视频生成链路：先合成高倍速音频并生成 ASS 字幕，再做混剪画面覆盖，最后烧录字幕。
7. 判断当前现象不是素材同步缺失，也不是字幕时间戳单点错误，而是混剪场景缺少对高倍速参数的体验约束。

## 当前判断

当前最可信的根因是：混剪任务允许 `voice_speed=3.0`，而混剪素材绑定文案较长，导致字幕和混剪覆盖区间被高倍速音频压缩到过短时间内。字幕可能仍然和音频同步，但观看者来不及读，体感就变成“字幕跟不上”。

普通视频没有混剪素材覆盖，也没有素材绑定文案窗口这层额外视觉负担，所以同样高倍速下问题不明显。

## 下一步建议

优先做小范围策略收口，不急着改字幕时间轴：

1. 在前端/CRM 层限制：选择混剪素材时，不展示或不允许提交 `3.0` 倍速。
2. 在后端 scheduler 层做兜底：存在 `Montage_dict` 或 `Photo_dict` 时，`voice_speed` 超过阈值就自动降级。
3. 阈值建议先用 `2.0`；如果产品更看重字幕可读性，可以直接用 `1.5`。
4. 增加日志字段：原始倍速、实际倍速、素材数量、素材文案长度、素材时长，方便复盘类似任务。
5. 增加回归测试，确保普通视频不受混剪倍速限制影响。
