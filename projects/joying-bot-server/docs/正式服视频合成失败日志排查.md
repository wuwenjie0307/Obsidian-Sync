# 正式服视频合成失败日志排查

## 适用场景

当前端或 CRM 反馈正式服出现以下问题时，优先按本文排查：

- 视频合成失败
- 视频生成失败
- 口型模型不工作
- 任务提交失败
- 任务超时
- 模型返回 `忙碌中`

注意：本文只适用于正式服旧口型生产链路，不适用于 h20 测试服或新模型 Docker 链路。

## 正式服信息

| 项目 | 值 |
|---|---|
| 服务器 | `222.71.55.27` |
| SSH 端口 | `9527` |
| SSH 用户 | `developer` |
| 生产目录 | `/data/project/prod_ai_autodone/` |
| supervisor 进程 | `ai_autodone_py_prod` |
| 应用日志 | `/data/project/prod_ai_autodone/logs/run.log` |
| 旧口型 Docker | `duix-avatar-*` 系列容器 |

不要把 h20 的日志路径当成正式服第一排查入口。例如 `/data/server_logs/supervisord/botserver_sch.out` 是 h20/测试链路常见日志，不是正式服旧口型问题的第一现场。

## 前端反馈时先要的信息

尽量让前端或 CRM 提供：

```text
失败时间
task_id
job_id
公司/项目名
用户或视频任务信息
前端看到的错误文案
```

最有用的是 `task_id`。拿到 `task_id` 后，直接 grep 全链路日志。

## 第一入口：正式服应用日志

登录正式服后进入生产目录：

```bash
ssh -p 9527 developer@222.71.55.27
cd /data/project/prod_ai_autodone
```

实时查看正式服视频生成应用日志：

```bash
tail -f /data/project/prod_ai_autodone/logs/run.log
```

按 `task_id` 查：

```bash
grep -n "替换成task_id" /data/project/prod_ai_autodone/logs/run.log | tail -120
```

不知道 `task_id` 时，按视频生成错误关键词查：

```bash
grep -nE "视频合成失败|任务提交失败|任务超时|忙碌中|easy/submit|Traceback|Exception|ERROR" \
  /data/project/prod_ai_autodone/logs/run.log | tail -200
```

当天实时日志通常在：

```text
/data/project/prod_ai_autodone/logs/run.log
```

历史日志可能按天轮转为：

```text
/data/project/prod_ai_autodone/logs/run.log.YYYY-MM-DD
```

查看有哪些日志文件：

```bash
ls -lh /data/project/prod_ai_autodone/logs/run.log*
```

查历史日志：

```bash
grep -nE "视频合成失败|任务提交失败|任务超时|忙碌中|easy/submit|Traceback|Exception|ERROR" \
```

## supervisor 辅助查看

正式服视频生成进程由 supervisor 管理：

```bash
sudo supervisorctl status ai_autodone_py_prod
```

查看 stdout：

```bash
sudo supervisorctl tail -f ai_autodone_py_prod stdout
```

日常定位业务链路优先看 `/data/project/prod_ai_autodone/logs/run.log`。`supervisorctl tail` 主要用于确认进程是否还在输出、是否刚重启、是否启动异常。

## 看到模型端口后再查 Docker

如果应用日志里出现类似：

```text
正在提交视频任务: http://127.0.0.1:6006/easy/submit
任务提交失败: {'code': 10001, 'data': {}, 'msg': '忙碌中', 'success': True}
```

说明 Bot 已经调到本机 `6006` 口型模型服务，接下来要查对应 Docker。

先定位端口和容器：

```bash
sudo docker ps | grep -E "6006|duix-avatar"
```

`6006` 当前记录对应：

```text
duix-avatar-gen-video-test4
```

查看容器日志：

```bash
sudo docker logs --tail=200 duix-avatar-gen-video-test4
```

实时查看：

```bash
sudo docker logs -f duix-avatar-gen-video-test4
```

如果日志里不是 `6006`，而是 `6002 / 6003 / 6007 / 6008 / 6004` 等端口，不要硬猜容器名，先用 `docker ps` 通过端口确认。

## 常见日志含义

### `视频合成失败：返回为空`

这是 Bot 业务层最终错误，不一定是根因。需要继续往前找同一个 `task_id` 的日志，看前面是否出现：

- `任务提交失败`
- `任务超时`
- `查询状态异常`
- `视频下载失败`
- `视频本地地址为空`

### `任务提交失败: {'code': 10001, 'msg': '忙碌中'}`

通常表示对应口型 Docker 认为自己仍在忙，常见原因：

- 上一个任务卡住
- busy 状态没有释放
- 容器内部队列异常
- 口型服务进程无响应但容器还活着

这时要查对应端口的 Docker 日志和状态。

### `任务超时! 已耗时: 7200s`

说明任务提交后 2 小时仍未完成。后续同端口任务继续返回 `忙碌中` 时，通常说明这个端口对应的口型容器已被卡住任务占用。

### `http://127.0.0.1:600X/easy/submit`

这是正式服 Bot 调本机旧口型 Docker 的接口。`600X` 是宿主机端口，需要通过 `docker ps` 映射到具体容器。

## 2026-06-04 案例：`6014` 被 `8405` 卡住导致 busy

### 现象

正式服出现“任务提交失败 / 忙碌中 / 视频合成失败”。本次不是 `ai_autodone_py_prod` 主进程挂掉，也不是当前队列持续堆积，而是旧口型 Docker 单实例被长任务占住。

### 快速结论

`6014` 对应容器 `duix-avatar-gen-video14`。任务 `8405` 从 `2026-06-04 10:48:37` 开始执行，直到 `14:48:42` 才触发 `task get result timeout, process will restart`，总耗时约 4 小时。期间新任务提交到同一个 `6014`，容器记录 `xxx -> busy`，上层就表现为“任务提交失败 / 忙碌中”。

### 本次证据

```text
2026-06-04 10:48:37  8405 -> current task start ...
2026-06-04 13:01:42  8457 -> busy
2026-06-04 13:12:30  8479 -> busy
2026-06-04 14:24:53  8494 -> busy
2026-06-04 14:47:04  8499 -> busy
2026-06-04 14:48:42  8405 -> task get result timeout, process will restart
2026-06-04 14:48:54  8405 -> 耗时:14426.513s
2026-06-04 14:52:25  8510 提交成功
2026-06-04 14:57:41  [8510]任务最终合成结果: /8510-r.mp4
2026-06-04 14:57:43  删除已完成任务: 8510
```

### 推荐排查命令

先确认生产主进程和当前业务队列：

```bash
sudo supervisorctl status ai_autodone_py_prod
tail -160 /data/project/prod_ai_autodone/logs/run.log
```

确认 `6014` 映射到哪个容器：

```bash
sudo docker ps --format '{{.Names}}\t{{.Status}}\t{{.Ports}}' | grep -E '6014|duix-avatar'
```

按时间窗看容器关键事件：

```bash
sudo docker logs --since '2026-06-04T10:45:00' --until '2026-06-04T15:00:30' duix-avatar-gen-video14 2>&1 \
  | grep -a -E 'POST /easy/submit|busy|任务最终合成结果|删除已完成任务|耗时|task get result timeout|GET /output|current task start'
```

如果不知道具体容器，先用 `run.log` 找到 `127.0.0.1:600X/easy/submit`，再用 `docker ps` 映射容器名。

### 处理建议

- 如果容器已经自行超时释放，并且后续任务已成功，不要盲目重启。
- 如果仍持续 busy，先确认当前占用任务 code 和占用时长，再评估是否只重启对应容器。
- 对 busy 期间失败的业务任务，需要结合 DB 状态决定是否人工重跑。
- 后续代码层面应把 busy 返回改成换实例或退避重试，减少单个旧口型容器卡住对业务任务的影响。

## 重要分流：先确认是不是 lucky-prod / 旧 prod_ai_autodone

2026-06-06 的谭慧两条“我的作品”视频失败证明：前端同样显示“视频合成阶段失败”，但不一定走旧 `/data/project/prod_ai_autodone` 链路。下次不要一上来就默认是 `6014 busy` 或旧口型 Docker，要先做服务线分流。

| 线索 | 旧口型 prod_ai_autodone | lucky-prod |
|---|---|---|
| 主要目录 | `/data/project/prod_ai_autodone` | `/data/projects/joying-bot-server-lucky-prod` |
| 主要日志 | `/data/project/prod_ai_autodone/logs/run.log*` | `/data/projects/joying-bot-server-lucky-prod/logs/run.log.YYYY-MM-DD` |
| 常见关键字 | `/easy/submit`, `忙碌中`, `任务提交失败`, `duix-avatar` | `[generate_video_task]`, `同步任务`, `异步预处理`, `视频任务重新合成`, `[处理任务-生成开始]` |
| DB 线索 | 旧 `ai_crm` 视频记录 | `zhugedata.t_video_generate_task` |
| 失败判断 | 看到 `127.0.0.1:600X/easy/submit` 后再查 Docker | 如果只看到同步/预处理/重置，没有 `[处理任务-生成开始]`，优先查 CRM 同步和 scheduler 取任务 |

快速判断：

```bash
# 旧链路没有 task/job 证据时，马上切到 lucky-prod 查
ls -lh /data/project/prod_ai_autodone/logs/run.log*
grep -nE "task_id|job_id|视频合成失败|任务提交失败|忙碌中|easy/submit" /data/project/prod_ai_autodone/logs/run.log* | tail -200

# lucky-prod 按 task_id/job_id 查当天滚动日志
grep -nE "task_id=替换|job_id=替换|接收到的task_id为替换|处理任务-生成开始|视频任务重新合成" \
  /data/projects/joying-bot-server-lucky-prod/logs/run.log.YYYY-MM-DD
```

只要没有 `/easy/submit`、没有 `忙碌中`、没有 `[处理任务-生成开始]`，就不要急着查或重启旧口型 Docker。这时更可能是状态同步、重跑接口、scheduler 查询条件或任务锁交接问题。
## 2026-06-06 案例：lucky-prod 任务 9396/9399 失败但未进入生成管线

### 现象

前端“我的作品”中用户谭慧两条视频失败，失败原因显示“视频合成阶段失败”：

| task_id | job_id | 标题 | 提交时间 CST | 失败/完成时间 CST |
|---:|---:|---|---:|---:|
| `9396` | `10078` | 捡漏状元府第泉中庭精装两房 | `2026-06-06 22:53:19` | `2026-06-06 23:03:51` |
| `9399` | `10081` | 状元府第一套全中庭的两房 | `2026-06-06 23:07:43` | `2026-06-06 23:11:15` |

### 快速结论

这次不是旧 `prod_ai_autodone` 的 `6014 busy` 问题。旧链路日志和旧库没有截图任务；真正相关的是 lucky-prod 的 `zhugedata.t_video_generate_task`。两条任务最终都是 `task_status=4`、`fail_reason=视频合成阶段失败`、`finish_time=0`、输出 URL 为空。

关键判断点：lucky-prod 日志只看到 CRM 同步、封面/描述/字幕轻量预处理，以及 `/video_generated_again` 重置日志；没有看到 `[处理任务-生成开始]`，也没有 `video_work.py`、`video_gen_service.py`、`/easy/submit`、`任务提交失败`、`忙碌中`。

因此它更像是 CRM 同步、重跑接口与 scheduler 取任务之间的状态/队列交接问题，不是模型 Docker 合成阶段真实开始后失败。

### 下次排查命令

先找 lucky 进程和实际 cwd：

```bash
ps -ef | grep -E "app_server_lucky_prod|app_server_lucky_sch" | grep -v grep
sudo readlink -f /proc/替换PID/cwd
```

查 lucky-prod 日志：

```bash
grep -nE "task_id=9396|job_id=10078|C1270T0U6062T20260606225319|接收到的task_id为9396|处理任务-生成开始|视频任务重新合成9396" \
  /data/projects/joying-bot-server-lucky-prod/logs/run.log.2026-06-06

grep -nE "task_id=9399|job_id=10081|C1270T0U6062T20260606230743|接收到的task_id为9399|处理任务-生成开始|视频任务重新合成9399" \
  /data/projects/joying-bot-server-lucky-prod/logs/run.log.2026-06-06
```

查 DB 时不要在笔记里记录密码；从服务器现有 config 读取连接信息即可。核心表是：

```sql
SELECT id, task_id, job_id, job_name, user_id, company_id, real_name,
       cover_title, task_status, progress, fail_reason, task_created_at,
       created_time, updated_time, finish_time, generate_video_url, video_source_url
FROM t_video_generate_task
WHERE task_id IN (9396, 9399) OR job_id IN (10078, 10081);

SELECT *
FROM t_video_material_template
WHERE job_id IN (10078, 10081);
```

### 本次踩坑记忆

- 不要因为 Obsidian 旧文档写着 `/data/project/prod_ai_autodone/logs/run.log`，就默认所有正式服“视频合成失败”都在旧链路。
- 旧库没有任务时，立刻转 lucky-prod；继续 grep 旧 `run.log` 会误导。
- `run.log` 要查带日期的滚动文件，不要只看当前文件。
- DB `created_time/updated_time` 和前端 CST 要做 +8 小时换算。
- “视频已重置”只说明本地 reset 接口执行过，不说明 scheduler 已重新开始生成。
- 缺少 `[处理任务-生成开始]` 是强信号：优先查状态同步/队列交接，而不是 Docker。
