# 正式服视频合成失败日志排查

## 适用场景

当前端或 CRM 反馈正式服出现以下问题时，优先按本文排查：

- 视频合成失败
- 视频生成失败
- 口型模型不工作
- 任务提交失败
- 任务超时
- 模型返回 `忙碌中`

注意：本文只适用于正式服旧口型生产链路，不适用于 h20 测试服或新模型 Docker 链路。

## 正式服信息

| 项目 | 值 |
|---|---|
| 服务器 | `222.71.55.27` |
| SSH 端口 | `9527` |
| SSH 用户 | `developer` |
| 生产目录 | `/data/project/prod_ai_autodone/` |
| supervisor 进程 | `ai_autodone_py_prod` |
| 应用日志 | `/data/project/prod_ai_autodone/logs/run.log` |
| 旧口型 Docker | `duix-avatar-*` 系列容器 |

不要把 h20 的日志路径当成正式服第一排查入口。例如 `/data/server_logs/supervisord/botserver_sch.out` 是 h20/测试链路常见日志，不是正式服旧口型问题的第一现场。

## 前端反馈时先要的信息

尽量让前端或 CRM 提供：

```text
失败时间
task_id
job_id
公司/项目名
用户或视频任务信息
前端看到的错误文案
```

最有用的是 `task_id`。拿到 `task_id` 后，直接 grep 全链路日志。

## 第一入口：正式服应用日志

登录正式服后进入生产目录：

```bash
ssh -p 9527 developer@222.71.55.27
cd /data/project/prod_ai_autodone
```

实时查看正式服视频生成应用日志：

```bash
tail -f /data/project/prod_ai_autodone/logs/run.log
```

按 `task_id` 查：

```bash
grep -n "替换成task_id" /data/project/prod_ai_autodone/logs/run.log | tail -120
```

不知道 `task_id` 时，按视频生成错误关键词查：

```bash
grep -nE "视频合成失败|任务提交失败|任务超时|忙碌中|easy/submit|Traceback|Exception|ERROR" \
  /data/project/prod_ai_autodone/logs/run.log | tail -200
```

当天实时日志通常在：

```text
/data/project/prod_ai_autodone/logs/run.log
```

历史日志可能按天轮转为：

```text
/data/project/prod_ai_autodone/logs/run.log.YYYY-MM-DD
```

查看有哪些日志文件：

```bash
ls -lh /data/project/prod_ai_autodone/logs/run.log*
```

查历史日志：

```bash
grep -nE "视频合成失败|任务提交失败|任务超时|忙碌中|easy/submit|Traceback|Exception|ERROR" \
  /data/project/prod_ai_autodone/logs/run.log* | tail -200
```

## supervisor 辅助查看

正式服视频生成进程由 supervisor 管理：

```bash
sudo supervisorctl status ai_autodone_py_prod
```

查看 stdout：

```bash
sudo supervisorctl tail -f ai_autodone_py_prod stdout
```

日常定位业务链路优先看 `/data/project/prod_ai_autodone/logs/run.log`。`supervisorctl tail` 主要用于确认进程是否还在输出、是否刚重启、是否启动异常。

## 看到模型端口后再查 Docker

如果应用日志里出现类似：

```text
正在提交视频任务: http://127.0.0.1:6006/easy/submit
任务提交失败: {'code': 10001, 'data': {}, 'msg': '忙碌中', 'success': True}
```

说明 Bot 已经调到本机 `6006` 口型模型服务，接下来要查对应 Docker。

先定位端口和容器：

```bash
sudo docker ps | grep -E "6006|duix-avatar"
```

`6006` 当前记录对应：

```text
duix-avatar-gen-video-test4
```

查看容器日志：

```bash
sudo docker logs --tail=200 duix-avatar-gen-video-test4
```

实时查看：

```bash
sudo docker logs -f duix-avatar-gen-video-test4
```

如果日志里不是 `6006`，而是 `6002 / 6003 / 6007 / 6008 / 6004` 等端口，不要硬猜容器名，先用 `docker ps` 通过端口确认。

## 常见日志含义

### `视频合成失败：返回为空`

这是 Bot 业务层最终错误，不一定是根因。需要继续往前找同一个 `task_id` 的日志，看前面是否出现：

- `任务提交失败`
- `任务超时`
- `查询状态异常`
- `视频下载失败`
- `视频本地地址为空`

### `任务提交失败: {'code': 10001, 'msg': '忙碌中'}`

通常表示对应口型 Docker 认为自己仍在忙，常见原因：

- 上一个任务卡住
- busy 状态没有释放
- 容器内部队列异常
- 口型服务进程无响应但容器还活着

这时要查对应端口的 Docker 日志和状态。

### `任务超时! 已耗时: 7200s`

说明任务提交后 2 小时仍未完成。后续同端口任务继续返回 `忙碌中` 时，通常说明这个端口对应的口型容器已被卡住任务占用。

### `http://127.0.0.1:600X/easy/submit`

这是正式服 Bot 调本机旧口型 Docker 的接口。`600X` 是宿主机端口，需要通过 `docker ps` 映射到具体容器。

## 2026-06-04 案例：`6014` 被 `8405` 卡住导致 busy

### 现象

正式服出现“任务提交失败 / 忙碌中 / 视频合成失败”。本次不是 `ai_autodone_py_prod` 主进程挂掉，也不是当前队列持续堆积，而是旧口型 Docker 单实例被长任务占住。

### 快速结论

`6014` 对应容器 `duix-avatar-gen-video14`。任务 `8405` 从 `2026-06-04 10:48:37` 开始执行，直到 `14:48:42` 才触发 `task get result timeout, process will restart`，总耗时约 4 小时。期间新任务提交到同一个 `6014`，容器记录 `xxx -> busy`，上层就表现为“任务提交失败 / 忙碌中”。

### 本次证据

```text
2026-06-04 10:48:37  8405 -> current task start ...
2026-06-04 13:01:42  8457 -> busy
2026-06-04 13:12:30  8479 -> busy
2026-06-04 14:24:53  8494 -> busy
2026-06-04 14:47:04  8499 -> busy
2026-06-04 14:48:42  8405 -> task get result timeout, process will restart
2026-06-04 14:48:54  8405 -> 耗时:14426.513s
2026-06-04 14:52:25  8510 提交成功
2026-06-04 14:57:41  [8510]任务最终合成结果: /8510-r.mp4
2026-06-04 14:57:43  删除已完成任务: 8510
```

### 推荐排查命令

先确认生产主进程和当前业务队列：

```bash
sudo supervisorctl status ai_autodone_py_prod
tail -160 /data/project/prod_ai_autodone/logs/run.log
```

确认 `6014` 映射到哪个容器：

```bash
sudo docker ps --format '{{.Names}}\t{{.Status}}\t{{.Ports}}' | grep -E '6014|duix-avatar'
```

按时间窗看容器关键事件：

```bash
sudo docker logs --since '2026-06-04T10:45:00' --until '2026-06-04T15:00:30' duix-avatar-gen-video14 2>&1 \
  | grep -a -E 'POST /easy/submit|busy|任务最终合成结果|删除已完成任务|耗时|task get result timeout|GET /output|current task start'
```

如果不知道具体容器，先用 `run.log` 找到 `127.0.0.1:600X/easy/submit`，再用 `docker ps` 映射容器名。

### 处理建议

- 如果容器已经自行超时释放，并且后续任务已成功，不要盲目重启。
- 如果仍持续 busy，先确认当前占用任务 code 和占用时长，再评估是否只重启对应容器。
- 对 busy 期间失败的业务任务，需要结合 DB 状态决定是否人工重跑。
- 后续代码层面应把 busy 返回改成换实例或退避重试，减少单个旧口型容器卡住对业务任务的影响。
