H20 侧 `VideoGenService.generate_video()` 的下游轮询超时阈值是 1800 秒。现场日志显示 2026-06-11 16:07:59 已经触发超时：

## 初步原因

这是两层问题叠加：

1. 下游 `duix-avatar-h20-test-6004` 内部卡在 20%。容器内存状态保持 `run / 20% / 视频特征提取完成 / result为空`，但子进程和 GPU 都无明显计算活动，判断为内部 worker 或结果产物等待逻辑挂起。
2. H20 侧虽然识别到了 1800 秒视频模型超时，但没有把任务和模型池锁可靠恢复。结果是任务留在 `task_status=2`，模型池配置留在 `is_active=2`，后续 scheduler 只扫描 `task_status=0`，因此不会再捞起这条任务。

## 影响范围

- 当前任务 1174 无法产出视频。
- 模型池配置 16 被长期占用，等价于少一个可用视频生成实例。
- 如果同类问题再次发生，会造成 `task_status=2 / is_active=2` 的挂起任务，测试服队列可能逐步被模型锁耗尽。

## 解决方向

代码修复优先级：

1. 确保 `VideoGenTimeoutError` 即使被 `StagePublicError` 包装，也能被上层识别为可重试的视频模型超时。
2. 超时后必须可靠保存任务状态：至少重置为 `task_status=0 / progress=0` 并写入可识别的 `fail_reason`，或明确标记失败并回调。
3. 包装层 `finally` 必须释放 `t_comfyui_config.is_active`，不能让模型池锁停留在 `2`。
4. 对 duix 返回长期 `status=1 / progress=20 / result为空` 的场景，保留 1800 秒总超时，同时考虑增加“进度无变化超时”的更早失败策略，避免空等 30 分钟。
5. 运维侧临时恢复可以重启 `duix-avatar-h20-test-6004`，并清理任务 1174 与 `config_id=16` 的挂起状态；执行前需要确认不会误伤其它任务。

## 代码修复记录

2026-06-11 已在 `joyingbot-new` 工作区完成代码层修复，核心目标是避免下游 duix / HeyGem 长时间卡住或超时后，把 H20 任务和模型池配置一起留在处理中状态。

- `router/service/video_server2/video_work.py`：`StagePublicError` 保留 `original_error`，并继续用异常链承接真实错误，避免 `VideoGenTimeoutError` 被阶段包装后丢失。
- `router/service/video_server/video_gen_service.py` 与 `router/service/video_server2/video_gen_service.py`：`is_video_model_busy_error()` 会递归识别 `original_error`、`original_exception`、`__cause__`、`__context__`，并识别 `video model busy` / `video model timeout` 文案。
- `scheduler/collect_scheduler.py`：`_process_single_video_task_with_config()` 外层异常兜底会写回任务状态；模型忙或超时回到 `task_status=0` 等待下轮调度，非可重试异常置为 `task_status=4`，避免停留在 `task_status=2`。
- `scheduler/collect_scheduler.py`：`generate_video_and_callback()` 在线程内二次校验后如果跳过任务，会释放预分配的 `config_id`，避免 `is_active=2` 泄漏。
